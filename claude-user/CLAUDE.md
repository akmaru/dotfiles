# Global Claude Code Instructions

リポジトリ横断で適用される共通の指示です。

## Communication Style

- **日本語で話す**: ユーザーへの返答は常に日本語で行う
- **簡潔に**: 必要な情報のみを伝える。無駄な説明や絵文字は使わない
- **技術的正確さを優先**: ユーザーの考えを無条件に肯定せず、客観的な技術情報を提供
- **明示的に依頼された場合のみ絵文字を使用**: デフォルトでは使用しない

## Coding Philosophy

- **シンプル第一**: 過度な抽象化を避け、必要最小限の実装を優先
- **YAGNI**: 現時点で必要な機能のみを実装
- **既存コードを尊重**: ファイルを読まずに変更提案しない
- **最小限の変更**: 要求された範囲のみを変更し、余計なリファクタリングや機能追加はしない
- **依頼外の変更は実装しない**: 機能追加、不要なヘルパー関数の作成、変更していないコードへのコメント・型注釈の追加などは行わない
- **リファクタリング・命名改善は提案にとどめる**: 気づいた場合は提案するが、承認なしに実装しない
- **エラーハンドリングはシステム境界のみ**: ユーザー入力・外部 API など。発生し得ないシナリオへのハンドリングは追加しない
- **後方互換性ハックはしない**: 未使用のものは `_` リネームせず完全に削除

## Security

- OWASP Top 10 の脆弱性（SQLインジェクション、XSS、コマンドインジェクションなど）に注意
- 不安全なコードを書いた場合は即座に修正
- `.env`, `credentials.json` などの機密ファイルのコミットを警告

## Git and Commits

コミットメッセージ形式: `[Category] Description`

- "why" を重視、"what" は diff で分かる
- プロジェクト固有の規約がある場合はそちらを優先

## Testing

- **機能追加**: 対応するテストケースを先に追加し、それが通過することを完了条件とする
- **バグ修正**: 既存テストで発見できなかった原因を調べ、テストケースの見直しまたはバグを再現するテストケースを追加してから修正する

## Documentation & File Operations

- コメントは「なぜ」を説明（「何を」はコードが説明する）
- README・ドキュメントファイルは必要に応じて作成・更新する
- 変更していないコードにドキュメントを追加しない
- 既存ファイルの編集を優先。シンボリックリンクを尊重

## Architecture Decision Records (ADR)

- 設計上の重要な決定や大きな変更には ADR を作成する
- 設計に影響する変更を行う際は、既存の ADR を確認し、決定・制約との矛盾があれば実装前にユーザーに報告する

## Tool Usage

- 専用ツールを優先: `cat` より `Read`、`sed` より `Edit`、`find` より `Glob`
- 複数の独立したツール呼び出しは並列実行
